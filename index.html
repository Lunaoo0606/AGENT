📋 一期产品方案全览

定位、核心功能、流程、迭代规划与关键原型

1 一期方案总览

一期不接入语义模型和数据模型，由人工确认分析范围、人工确认分析计划，聚焦搭建 Agent 的运行骨架、交付能力和沉淀与自动化。

2 核心功能（四维）

产品围绕四个维度构建，从「用户看得懂」到「系统跑得好」逐层递进。

🔄
过程透明
让用户看到 Agent 在做什么、为什么这样做
⚡
技能
将分析经验沉淀为可复用的结构化能力
🤖
自动化
让分析定期运行、异常自动触发、结果自动推送
🛡️
质量保障
约束 + 评测 + 反馈闭环，持续驱动分析质量提升
维度解决什么问题一期（ABC 三个迭代）做完后能实现什么
过程透明用户不信任黑箱结果简单/复杂自动分流（用户可切换）；分析计划可查看/确认；分析过程实时可见；交付物三段式 UI呈现（Chat + Process + Artifacts）
技能每次分析都从零开始，无法积累预制技能 + 用户手选/语义自动匹配调用；从现有报告沉淀技能；技能中心（系统+个人）浏览搜索；可视化技能编辑器
自动化重复性分析需要人工触发创建自动化任务；定时执行 + 阈值触发（指标越线自动归因）；多渠道推送执行结果
质量保障分析质量无法量化和系统性改进基础约束 + 评测驱动的规则迭代 ；全量可观测埋点 -> LLM-as-Judge 自动评测标注bad case；根据评测结果持续改进模型约束

8.2 分析计划

9 技能系统

技能定义「做什么分析、怎么做、输出约束是什么」。技能 = 一套结构化的 function calling schema，告诉模型可以调用哪些分析能力、需要什么参数、输出须满足什么约束。

9.1 技能结构定义

技能结构 = 名称 + 描述 + 分析步骤描述 + 参数定义 + 约束

字段类型必填说明
skill_idstring是唯一标识，如 conversion_attribution
namestring是名称。展示在技能选择器中。如「保养机油转化分析」
descriptionstring是描述。技能功能描述，B 期自动匹配的核心依据。包含：适用场景 + 分析目标 + 典型问法
analysis_stepsobject[]是分析步骤描述。有序步骤列表，每步包含 step_description（做什么）+ method（分析方法标签）。模型据此生成分析计划
parametersobject否参数定义。技能需要的输入参数（JSON Schema 格式）。包括拆解维度、对比周期、异常阈值等分析策略类参数。每个参数含 type、description、default
constraintsstring[]否约束/限定。输出约束规则，如「必须量化贡献占比」「必须给出行动建议」「异常定义：环比 >10%」
validation_rulesobject[]否技能级校验规则。与系统指令中的全局约束互补：全局约束对所有分析生效（如「结论必须包含行动建议」），validation_rules 定义该技能特有的校验逻辑（如「归因分析必须覆盖 Top3 维度」）。A/B 期由人工评测驱动补充，C 期供系统代码自动校验使用
versionstring是技能版本号，用于追踪和评测

技能示例：保养机油转化分析

字段内容
名称保养机油转化分析
描述分析转化率变化原因，按流量来源拆解并下钻定位异常。适用于「转化率为什么下降了」「哪个渠道转化有问题」
分析步骤① 确认整体转化率下降数据 → ② 按流量来源拆解 Top 5 → ③ 异常来源下钻 → ④ 计算各来源 UV 价值 → ⑤ 结论与建议
参数报警指标（默认：端转）、拆解维度（默认：流量来源）、过滤条件、时间范围（默认：近 3 月）、异常阈值（默认：10%）
约束必须给环比数值、必须量化贡献占比、必须给行动建议、异常定义：环比 >10%
校验规则归因分析须覆盖 Top3 维度、拆解占比之和须 = 100%（技能特有规则，与全局约束互补）

9.2 参数提取机制

用户选定技能后，模型需要从用户的自然语言问题中提取参数值填入技能的 parameters。

参数提取流程

步骤说明示例
1. 用户提问自然语言问题 + 已选技能「帮我看下上周各城市的 GMV 变化」+ 已选「周报 GMV 归因」
2. 模型提取模型根据 function schema 中各参数的 description，从用户问题中提取对应值识别 focus_metric=GMV, breakdown_dims=[城市], compare_period=周环比
3. 默认填充用户未明确提及的参数，使用 schema 中的 default 值anomaly_threshold 未提及 → 使用默认值 10
4. 上下文补充部分参数可从数据上下文中推断数据中有 city、category 列 → 推断可用拆解维度
5. 展示确认提取的参数展示在分析计划中，用户可在确认模式下修改分析计划卡片中列出所有参数及其值

参数来源优先级

优先级来源说明
1（最高）用户显式指定用户在问题中明确提到的参数值，如「按城市维度」
2技能默认值技能 parameters 中定义的 default 值
3（最低）数据上下文推断从数据 Schema 中推断，如数据中有 city 列 → 可用于拆解
参数提取的质量评测 评测集中每个 Case 都包含 expected_params 字段（见第 11 节），用于量化参数提取准确率。准确率 = 正确提取的参数数 / 技能总参数数。迭代 A 目标：手选技能场景下参数准确率 ≥ 80%。

9.4 从现有报告沉淀技能（迭代 B）

业务团队已有大量周报/月报等分析报告，其分析逻辑可以反向提取为技能定义，降低技能创建门槛。

步骤操作产出
1. 上传报告用户在技能中心点击「从现有报告沉淀技能」，上传一份或多份已有的分析报告（Excel/PDF/截图）报告内容
2. 结构提取模型自动解析报告，识别分析步骤（数据来源 → 计算逻辑 → 拆解维度 → 结论模式）和隐含参数技能草稿（含名称、描述、步骤、参数、约束）
3. 人工校审用户在技能编辑器中审核和修改草稿，补充模型未识别的业务规则完整技能定义
4. 验证测试用户用真实问题测试新技能，确认输出符合预期后发布可用技能

11.0 三层质量机制总览

层次名称什么时候运行谁来做解决什么问题迭代
1 运行时防护 每步执行后，实时 系统代码（确定性逻辑，不调用大模型） 快速拦截明显错误：数据范围越界、步骤数过多、数值不一致 A

11.1 约束规则体系

A 期：预制基础约束（安全相关）；B 期：而是通过评测积累数据、迭代 Prompt 规则。 C 期：补充自动校验逻辑

规则存放位置说明

不同类型的规则存放在不同位置，由系统代码在运行时统一加载和执行：

规则类型存放位置执行方式示例
安全规则系统配置（代码级）系统代码强制执行，大模型无法绕过沙箱禁止外网访问、数据按权限过滤

C 期自动校验（随自动化上线）

当分析由定时任务或阈值触发自动执行时，没有用户实时查看结果。此时需要系统代码在关键节点做规则检查：

校验点校验内容不通过处理
